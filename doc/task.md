
# 📋 大厂入职实战：TechConnect —— 开发者社区内容系统 (MVP)
## 1. 项目背景
我们需要为公司内部开发一个类似“技术朋友圈”的轻量级社交平台。用户可以发布技术动态、关注他人、并针对动态进行点赞和评论。
**核心考点**：复杂关系型数据库建模、高频互动的性能处理、无限滚动列表的极致体验。
## 2. 核心技术栈要求
* **前端**：必须使用现代框架（推荐 **React** 或 **Vue 3**），配合 **TypeScript**。
* **后端**：使用 **Node.js** (如 Express, NestJS 或 Fastify)。
* **数据库**：必须使用**关系型数据库** (如 **PostgreSQL** 或 **MySQL**)。
* **ORM**：建议使用 Prisma 或 TypeORM（展示你对工程化工具的掌握）。
---
## 3. 详细功能需求清单
### P0：基础建设（必须完成）
* **鉴权系统**：
* 用户注册/登录（密码需加盐哈希存储）。
* 基于 **JWT** 的身份验证，支持登录态持久化。
* **动态 (Post) CRUD**：
* 发布动态：支持文本内容，并模拟多图上传（后端存储图片 URL）。
* 删除动态：仅限作者本人操作。
* **核心 Feed 流**：
* **分页加载**：实现后端分页（建议使用 Cursor-based pagination 以应对高频更新）。
* **无限滚动**：前端实现列表滚动触底自动加载下一页。
### P1：社交互动（体现业务深度）
* **点赞系统 (Like)**：
* 同一用户对同一动态只能点赞一次。
* **乐观更新**：前端点击点赞后 UI 立即变化，异步向后端发请求。
* **评论系统 (Comment)**：
* 支持对动态进行一级评论（暂不要求嵌套回复）。
* 显示评论列表，按时间倒序排列。
* **社交关系 (Follow)**：
* 支持“关注”和“取关”功能。
* 用户个人中心需展示：粉丝数、关注数、发布的动态数。
### P2：工程优化（加分项/高手项）
* **数据一致性**：在关系型数据库中，如何高效维护动态的点赞总数（实时 COUNT 还是 字段冗余？需在 README 解释理由）。
* **响应式设计**：完美适配手机端浏览器和桌面端。
* **简单的搜索**：支持按关键词搜索动态内容。
---
## 4. 交付要求与判分标准
### 交付物
1. **代码仓库**：包含完整的 `frontend` 和 `backend` 代码。
2. **README.md**：
* 如何本地运行（含数据库初始化步骤）。
* **数据库 ER 图**（说明表与表之间的关系）。
* 你的技术选型理由及遇到的最大挑战。
3. **API 文档**：简单的接口列表说明。
### 面试官如何打分？
* **数据库设计 (30%)**：表结构是否符合范式？索引是否合理？是否处理了用户-动态-点赞的多对多关系？
* **前端性能 (25%)**：列表滚动是否卡顿？组件拆分是否解耦？TS 类型定义是否严谨？
* **业务鲁棒性 (25%)**：是否处理了网络报错（点赞失败怎么办）？是否存在越权操作（修改他人动态）？
* **工程化能力 (20%)**：代码风格是否一致？是否有清晰的项目目录结构？
---
## 5. 出题人的疑问 (面试官提问环节)
作为一个 Junior 开发，当你拿到这份需求时，我作为“面试官”会先听听你的初步构思。请回答以下几个关键点：
1. **数据库建模**：在你的设计中，`用户(User)`、`动态(Post)` 和 `点赞(Like)` 这三张表是如何关联的？
2. **性能瓶颈**：如果一个用户关注了 500 人，他在首页刷新 Feed 流时，你的 SQL 语句打算怎么写？（是直接 `JOIN` 还是分两次查询？）
3. **框架选择**：针对这个项目，你打算选择哪一个具体的前端状态管理方案（Redux, Zustand, 还是简单的 Context API）？为什么？

# 验收

## 功能验收
```
1. 注册 → 登录 → 看到 Feed
2. 发一条动态 → 出现在 Feed 顶部
3. 滚动到底部 → 自动加载下一批（无限滚动）
4. 点赞 → UI 立即变化 → 刷新后数量正确
5. 关注一个用户 → 切换到关注流 → 只看到他的动态
6. 访问个人主页 → 显示粉丝数 / 关注数 / 动态数
```
